# Observability

llmstack includes built-in observability with Prometheus for metrics collection and Grafana for dashboarding. When enabled, a pre-configured dashboard gives you immediate visibility into your LLM stack's performance.

## Enabling Observability

Observability is enabled by default. To disable it, set `observe.metrics: false` in `llmstack.yaml`:

```yaml
observe:
  metrics: true          # Set to false to skip Prometheus + Grafana
  dashboard_port: 8080   # Grafana port
  retention: 7d          # How long Prometheus retains data
```

## Accessing the Dashboard

After running `llmstack up`, open Grafana at:

```
http://localhost:8080
```

Default credentials:

| Field | Value |
|---|---|
| Username | `admin` |
| Password | `llmstack` |

Anonymous read-only access is also enabled, so you can view dashboards without logging in.

## Metrics Collected

The gateway exposes a `/metrics` endpoint in Prometheus exposition format. The following metrics are available:

### Request Metrics

| Metric | Type | Description |
|---|---|---|
| `llmstack_requests_total` | Counter | Total requests by method, path, and status code |
| `llmstack_request_duration_seconds` | Histogram | Request latency (p50, p90, p99) |
| `llmstack_requests_in_progress` | Gauge | Currently active requests |

### Token Metrics

| Metric | Type | Description |
|---|---|---|
| `llmstack_tokens_input_total` | Counter | Total input tokens processed |
| `llmstack_tokens_output_total` | Counter | Total output tokens generated |

### Cache Metrics

| Metric | Type | Description |
|---|---|---|
| `llmstack_cache_hits_total` | Counter | Number of cache hits |
| `llmstack_cache_misses_total` | Counter | Number of cache misses |

### Rate Limiter Metrics

| Metric | Type | Description |
|---|---|---|
| `llmstack_rate_limit_rejections_total` | Counter | Requests rejected by rate limiter |

### Circuit Breaker Metrics

| Metric | Type | Description |
|---|---|---|
| `llmstack_circuit_breaker_state` | Gauge | Current state (0=closed, 1=open, 2=half_open) |
| `llmstack_circuit_breaker_failures` | Counter | Total backend failures |
| `llmstack_circuit_breaker_rejections` | Counter | Requests rejected by open circuit |

## Dashboard Panels

The pre-provisioned Grafana dashboard includes these panels:

### Request Rate

A time series chart showing requests per second, broken down by endpoint. Useful for understanding traffic patterns and identifying sudden spikes.

### Latency (p50 / p99)

Histogram panels showing the 50th and 99th percentile latency for each endpoint. The p99 is particularly important -- if it spikes, some users are experiencing significantly slower responses.

### Token Throughput

A stacked area chart showing input and output tokens per second. This tells you how hard the inference backend is working.

### Error Rate

A time series of 4xx and 5xx responses. A spike in 5xx errors usually means the inference backend is struggling. A spike in 4xx errors may indicate authentication issues or rate limiting.

### Cache Hit Rate

A gauge showing the percentage of requests served from cache. A higher hit rate means lower latency and less load on the inference backend.

### Circuit Breaker State

A state timeline showing when the circuit breaker is CLOSED (healthy), OPEN (backend down), or HALF_OPEN (testing recovery).

### Rate Limit Rejections

A counter of requests rejected by the rate limiter, broken down by client. Useful for identifying abusive clients.

## Prometheus Configuration

Prometheus is configured to scrape the gateway's `/metrics` endpoint every 15 seconds. The configuration is automatically generated by llmstack and mounted into the Prometheus container.

You can access Prometheus directly at `http://localhost:9090` to run ad-hoc PromQL queries.

### Example Queries

Request rate over the last 5 minutes:

```promql
rate(llmstack_requests_total[5m])
```

p99 latency:

```promql
histogram_quantile(0.99, rate(llmstack_request_duration_seconds_bucket[5m]))
```

Cache hit ratio:

```promql
rate(llmstack_cache_hits_total[5m]) /
(rate(llmstack_cache_hits_total[5m]) + rate(llmstack_cache_misses_total[5m]))
```

## Custom Alerts

You can add custom alerting rules to Prometheus. Create a rules file and mount it into the Prometheus container using `llmstack export` to generate a `docker-compose.yml`, then customize the Prometheus volumes.

Example alert rule for high error rate:

```yaml
groups:
  - name: llmstack
    rules:
      - alert: HighErrorRate
        expr: rate(llmstack_requests_total{status=~"5.."}[5m]) > 0.1
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High 5xx error rate detected"
```

## Integration with External Systems

### Existing Prometheus

If you already run Prometheus, you can scrape the gateway directly without the bundled Prometheus. Add this to your Prometheus config:

```yaml
scrape_configs:
  - job_name: llmstack
    scrape_interval: 15s
    static_configs:
      - targets: ["localhost:8000"]
    metrics_path: /metrics
```

Then disable the built-in observability stack:

```yaml
observe:
  metrics: false
```

### Log Aggregation

The gateway outputs structured JSON logs, which are compatible with any log aggregation system (ELK, Loki, Datadog, etc.). Use `llmstack logs gateway` to view logs directly, or configure your log collector to read from the Docker container.
